## 1. 产品背景与目标

### 1.1 业务背景

高校AI领域招投标信息分散在多个政府采购平台（中国政府采购网、各省市采购网、招标投标公共服务平台等），手动检索、筛选和分析这些信息耗时巨大且容易遗漏。市场分析人员需要一个自动化工具来：

- 跨平台采集与"AI+高校"相关的招投标公告
- 智能筛选出真正与高校AI建设相关的项目
- 利用AI自动从公告原文中提取关键结构化信息

### 1.2 产品目标

| 目标      | 衡量标准                        |
| ------- | --------------------------- |
| 自动化采集   | 覆盖 ≥3 个招投标信息源，支持关键词与时间范围搜索  |
| 精准筛选    | 通过高校关键词漏斗，从全量AI招标中筛选出高校相关项目 |
| AI结构化分析 | 自动从公告原文提取12个关键字段，准确率 ≥85%   |
| 高效处理    | 支持多线程并发分析，处理速度 ≥5条/秒        |
| 结果可导出   | 支持 JSONL、Excel、CSV 多种输出格式   |

### 1.3 目标用户

- **主要用户**：市场分析人员 / 售前工程师 — 需了解高校AI采购趋势和竞品信息
- **次要用户**：管理层 — 查看汇总报表了解市场动态

## 2. 系统架构

### 2.1 两阶段流水线

```
┌─────────────────────────────────────────────────────────────┐
│                     第一阶段：数据采集                         │
│                                                             │
│  信源发现 → 策略制定 → 自动化采集 → 数据清洗 → 结果输出        │
│                                                             │
│  输入：关键词 + 时间范围 + 高校过滤词                          │
│  输出：结构化的招投标公告列表 (JSONL / Excel)                  │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           ▼
              ┌─────────────────────────┐
              │   第一阶段结果前端展示     │  ← 用户可查看所有采集结果
              │   用户审核 & 决策是否继续  │  ← 用户手动触发下一阶段
              └───────────┬─────────────┘
                          │ (用户确认后)
                          ▼
┌─────────────────────────────────────────────────────────────┐
│                     第二阶段：AI分析                          │
│                                                             │
│  逐条抓取公告原文 → AI提取结构化字段 → 结果汇总输出            │
│                                                             │
│  输入：第一阶段产出的 URL 列表（用户可勾选筛选）               │
│  输出：带AI分析结果的增强数据 (JSONL / Excel)                 │
└─────────────────────────────────────────────────────────────┘
```

**关键设计原则**：

- 第一阶段和第二阶段的衔接**不作为自动流程**，中间插入用户审核节点
- 每个阶段的主要节点状态（采集进度、已采集数量、过滤结果、分析进度等）在前端实时可见
- 第二阶段的启动由用户**手动决策**，系统不自动触发

### 2.2 技术栈

| 层级   | 技术选型                                      |
| ---- | ----------------------------------------- |
| 后端框架 | Python + FastAPI                          |
| 前端框架 | React + TypeScript                        |
| 实时通信 | WebSocket / SSE（状态推送）                     |
| 爬虫引擎 | Python + requests + BeautifulSoup4        |
| 数据层  | JSONL (缓存/中间数据), Excel/CSV (最终输出), pandas |
| AI引擎 | DeepSeek Chat API / Dify 工作流              |
| 并发框架 | concurrent.futures (ThreadPoolExecutor)   |
| 架构模式 | 策略模式 (Strategy Pattern) — 爬虫可插拔扩展         |

## 3. 功能需求

### 3.1 第一阶段：数据采集

#### F1.1 多信源爬虫框架

**描述**：基于策略模式的泛化爬虫框架，支持通过配置切换不同的招投标网站。

**核心抽象**：

- `BaseSearchStrategy`：定义搜索策略的标准接口
  - `build_search_url(page)` — 构建搜索URL
  - `build_search_params(page)` — 构建搜索参数
  - `parse_results(html)` — 解析搜索结果
  - `detect_blocked(html)` — 反爬检测
  - `has_more_pages(html, page)` — 分页判断

**已实现的策略**：

| 策略名称                       | 信源网站                                    | 状态       |
| -------------------------- | --------------------------------------- | -------- |
| CCGPSearchStrategy         | 中国政府采购网 (search.ccgp.gov.cn)            | 已实现      |
| CTBPPSearchStrategy        | 招标投标公共服务平台 (bulletin.cebpubservice.com) | 框架已有，待完善 |
| ChinaBiddingSearchStrategy | 中国招标网 (chinabidding.cn)                 | 框架已有，待完善 |

**验收标准**：

- [ ] 任意策略实现 `BaseSearchStrategy` 接口即可接入爬虫
- [ ] 新增一个信源只需编写一个策略类文件（<200行代码）
- [ ] 爬虫自动处理缓存去重、失败重试（3次）、反爬延迟

#### F1.2 关键词过滤漏斗

**描述**：采用两层关键词过滤机制，从海量招标信息中精准定位高校AI相关项目。

- **第一层（信源搜索关键词）**：在招投标网站搜索框中使用的关键词，如 "AI"、"人工智能"、"大模型"、"智能" 等，用户可在前端自由配置
- **第二层（高校过滤关键词）**：对搜索结果进行二次筛选，匹配高校相关实体。用户在 Web 前端自由编辑和管理过滤关键词列表（支持增删改），系统默认提供推荐词库（如 `['大学', '学院', '高职', '职业技术', '师范', '理工', '学校']`）作为初始值

**前端交互**：

- 提供过滤关键词管理面板，支持用户新增、删除、启用/禁用单个关键词
- 支持批量导入/导出关键词列表
- 关键词变更后即时生效，无需重启采集任务
- 用户可保存多组过滤方案，按场景快速切换（如"本科院校过滤""高职院校过滤"等）

**验收标准**：

- [ ] 支持配置多组搜索关键词 + 过滤关键词
- [ ] 过滤在采集阶段实时进行，减少无效存储
- [ ] 过滤关键词通过前端界面自由编辑，不再硬编码在源码中
- [ ] 前端修改关键词后实时同步至爬虫引擎

#### F1.3 数据标准化

**描述**：采集到的招投标信息统一为 `TenderItem` 结构：

| 字段                   | 说明     | 示例             |
| -------------------- | ------ | -------------- |
| 标题 (title)           | 公告标题   | XX大学AI教学平台采购项目 |
| URL                  | 公告详情链接 | http\://...    |
| 发布日期 (publish\_date) | 公告发布日期 | 2025-09-15     |
| 采购人 (purchaser)      | 采购单位名称 | XX大学           |
| 代理机构 (agency)        | 招标代理机构 | XX招标有限公司       |
| 所在区域 (area)          | 项目所属地区 | XX省XX市         |
| 来源 (source)          | 信源名称   | 中国政府采购网        |

**验收标准**：

- [ ] JSONL 格式缓存，每行一条记录
- [ ] 支持按URL自动去重
- [ ] 支持断点续采（缓存加载）

#### F1.4 缓存与断点续采

**描述**：采用 JSONL 格式的缓存文件，每条采集到的记录实时追加写入。重新启动时自动加载已有缓存，跳过已采集的URL。

**验收标准**：

- [ ] 中断后重启不重复采集
- [ ] 缓存文件可独立使用（不依赖运行状态）
- [ ] 支持指定起始页继续采集

#### F1.5 流水线状态可见性与用户决策节点

**描述**：整个采集-分析流水线中，各关键节点的状态必须在前端显式呈现，用户对流程推进拥有完整的知情权和决策权。

**核心状态节点**：

| 节点       | 状态展示内容                 | 用户可执行操作                   |
| -------- | ---------------------- | ------------------------- |
| 搜索采集进度   | 当前信源、已搜索页数、已获取条目数、速率   | 暂停/继续/停止采集                |
| 高校过滤结果   | 过滤前总数、过滤后数量、被过滤条目列表    | 查看被过滤条目、调整过滤关键词后重新过滤      |
| 第一阶段完成   | 最终采集结果列表（标题、采购人、日期、来源） | 逐条查看详情、勾选/反选条目、导出当前结果     |
| 第二阶段启动决策 | 待分析条目数量、预计耗时           | **手动触发第二阶段**，支持全选或仅分析勾选条目 |
| AI分析进度   | 已分析/总数、成功/失败数、当前处理条目   | 暂停/继续/停止分析                |
| 第二阶段完成   | 分析结果汇总（成功率、各状态分布）      | 查看单条分析详情、导出最终结果           |

**设计原则**：

- 第一阶段采集的所有结果**必须先展示在前端**，让用户感知实际采集到了什么数据，之后才能进入第二阶段
- 第二阶段（AI分析）**不自动启动**，必须由用户在前端手动触发
- 用户可在第一阶段结果中**选择性勾选**需要分析的条目，未勾选的条目不进入AI分析
- 每个阶段的进度和中间结果实时通过 WebSocket/SSE 推送到前端

**验收标准**：

- [ ] 第一阶段采集进度实时可见（当前页数、已采集数）
- [ ] 第一阶段完成后，所有采集结果呈现在前端列表中
- [ ] 用户可勾选/反选条目，选择哪些条目进入第二阶段
- [ ] 第二阶段必须由用户手动点击触发，不可自动开始
- [ ] AI分析进度实时可见（已完成数/总数，成功/失败数）
- [ ] 每个阶段的状态切换有明确的前端状态指示

### 3.2 第二阶段：AI分析

#### F2.1 公告原文抓取

**描述**：根据用户在 Web 前端勾选的第一阶段产出URL列表，逐个访问招投标公告详情页，抓取并清洗HTML内容为纯文本。

**处理规则**：

- 去除 `<script>`、`<style>` 标签
- 去除HTML标签，保留纯文本
- 压缩多余空白字符
- 单篇最长截取 8000 字符

**验收标准**：

- [ ] 单篇抓取超时时间 30 秒
- [ ] 抓取失败时记录错误信息，不阻塞后续处理

#### F2.2 AI结构化分析

**描述**：调用大模型API，从公告原文中提取12个结构化字段。

**输出字段定义**：

| 字段名                     | 说明                                      | 必填 |
| ----------------------- | --------------------------------------- | -- |
| `project_status`        | 项目状态（招标中/已中标/已废标/资格预审）                  | 否  |
| `tender_release_date`   | 招标公告发布日期                                | 否  |
| `bid_award_date`        | 中标公告发布日期                                | 否  |
| `purchasing_entity`     | 采购方全称（通常为高校名称）                          | 否  |
| `project_name`          | 项目官方全称                                  | 否  |
| `purchaser_info`        | 采购人联系方式/地址                              | 否  |
| `product_type`          | **核心字段**：采购产品/服务详情（不得使用"货物类""服务类"等笼统词汇） | 是  |
| `budget_amount`         | 预算金额                                    | 否  |
| `winning_bid_amount`    | 中标金额                                    | 否  |
| `supplier_name`         | 中标供应商全称                                 | 否  |
| `procurement_type`      | 采购类型（公开招标/邀请招标/竞争性谈判/单一来源等）             | 否  |
| `tender_notice_url`     | 招标公告链接                                  | 是  |
| `procurement_documents` | 采购文件信息（文件名/下载链接）                        | 否  |

**AI调用配置**：

- 模型：DeepSeek Chat 或 Dify 工作流
- 温度参数：0.1（确保输出稳定）
- 超时时间：120 秒
- 输出格式：严格JSON，无额外文本

**验收标准**：

- [ ] AI返回结果必须为合法JSON对象
- [ ] `product_type` 字段不得出现"货物类""服务类""工程类"等笼统词
- [ ] 原文未提及的字段值设为 `null`
- [ ] API调用失败时记录错误，继续处理下一条

#### F2.3 多线程并发处理

**描述**：支持多线程并发抓取和分析，大幅提升第二阶段处理效率。

**配置参数**：

- `max_workers`：并发线程数（默认10）
- `delay`：任务提交间隔（默认1.5秒）

**验收标准**：

- [ ] 支持单线程/多线程模式切换
- [ ] 多线程场景下结果文件写入线程安全
- [ ] 支持断点续处理（检测已处理索引，跳过已完成记录）
- [ ] 进度实时输出到控制台

#### F2.4 结果输出

**描述**：分析结果以 JSONL 格式保存，每条记录包含原始数据和AI分析结果。

**输出结构**：

```json
{
  "index": 0,
  "original": { "标题": "...", "URL": "...", ... },
  "analysis": {
    "success": true,
    "data": {
      "project_status": "...",
      "product_type": "...",
      ...
    }
  }
}
```

**验收标准**：

- [ ] 支持 JSONL 格式（增量写入，适合流水线）
- [ ] 支持 Excel 格式（最终交付，包含关键字段）
- [ ] 成功/失败数量统计

## 4. 非功能需求

### 4.1 性能

| 指标         | 目标值              |
| ---------- | ---------------- |
| 第一阶段单页采集耗时 | ≤30秒（含反爬延迟）      |
| 第二阶段单条分析耗时 | ≤30秒（含网络抓取+AI调用） |
| 多线程并发吞吐量   | ≥10条/分钟（10线程）    |
| 缓存文件最大容量   | 无硬限制（JSONL追加写入）  |

### 4.2 可靠性

- **反爬保护**：自动检测"访问受限""频繁访问"关键字，触发后停止当前信源的采集
- **失败重试**：单页请求失败自动重试3次，间隔10秒
- **API异常处理**：AI调用失败不中断流水线，记录错误继续处理
- **去重保证**：
  - 采集阶段：URL 唯一性检查（内存 + 缓存双层去重）
  - 输出阶段：Excel导出时 DataFrame.drop\_duplicates 兜底

### 4.3 可扩展性

- **信源扩展**：新增目标网站只需实现 `BaseSearchStrategy` 抽象类
- **分析字段扩展**：修改 AI Prompt 中的字段定义即可
- **输出格式扩展**：添加新的 `save_to_*` 方法即可

### 4.4 安全性

- API密钥不应硬编码在源码中（当前版本存在此问题，建议后续迁移至环境变量）
- 遵守目标网站的 robots.txt 和访问频率限制

### 4.5 易用性

- 命令行参数控制：处理数量、请求间隔、并发数、单/多线程模式
- 清晰的日志输出：实时显示进度、成功/失败统计
- 中文界面和提示信息

## 7. 已知问题与改进方向

### 7.1 当前限制

| 问题                                      | 影响               | 优先级 |
| --------------------------------------- | ---------------- | --- |
| API密钥硬编码在源码中                            | 安全风险             | 高   |
| 部分信源策略未完善（CTBPP/ChinaBidding）           | 信源覆盖不完整          | 中   |
| 无Web管理界面（当前为CLI工具）                      | 使用门槛较高，已纳入v1.1规划 | 高   |
| 无定时调度机制                                 | 需手动运行            | 低   |
| `tender_notice_url` 字段来自爬虫URL而非公告详情页URL | 数据准确性            | 中   |
| 无数据可视化/报表功能                             | 分析洞察不直观          | 低   |
| 缓存文件为本地JSONL，无数据库支持                     | 大规模数据管理困难        | 低   |

### 7.2 后续规划

**v1.1 — 安全与健壮性**

- API密钥迁移至环境变量或配置文件
- 完善 CTBPP 和 ChinaBidding 策略的页面解析逻辑
- 用户自定义过滤方案持久化保存

**v1.2 — 自动化增强**

- 添加定时调度（每日/每周自动运行）
- 增量更新机制（仅采集上次运行后的新数据）
- 采集任务历史记录与回溯

**v2.0 — 数据化**

- 数据库存储（SQLite/PostgreSQL），替代纯 JSONL 文件
- 竞品分析报表（市场趋势、采购热点、供应商排名）
- 数据可视化看板（词云图、趋势折线图、区域热力图）
- 多用户支持与权限管理

## 8. 术语表

| 术语            | 说明                             |
| ------------- | ------------------------------ |
| 信源            | 招投标信息发布的源网站，如中国政府采购网           |
| 策略 (Strategy) | 针对特定信源的搜索实现，包含URL构建、参数组装、结果解析等 |
| 漏斗过滤          | 通过两层关键词筛选从全量数据中定位目标项目的过程       |
| TenderItem    | 标准化招投标数据结构，包含标题、URL、采购人等字段     |
| 缓存 (Cache)    | JSONL格式的中间文件，防止重复采集，支持断点续采     |
| Dify          | 一个AI工作流平台，本系统通过其API调用大模型进行分析   |
